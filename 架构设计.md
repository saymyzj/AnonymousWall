# AnonymousWall — 架构设计文档

## 一、技术选型总览

| 层级 | 技术选择 | 版本 | 选型理由 |
|------|---------|------|---------|
| **前端框架** | Vue 3 + TypeScript | Vue 3.4+ | 组合式 API + TS 类型安全，学习曲线友好，适合中等复杂度项目 |
| **前端构建** | Vite | 5.x | 极速 HMR，开箱即用支持 Vue 3 + TS |
| **UI 组件库** | Vant 4（移动端）+ Naive UI（管理后台） | - | Vant 移动端体验优秀；Naive UI 对 Vue 3 + TS 支持一流，适合后台 |
| **状态管理** | Pinia | 2.x | Vue 3 官方推荐，TS 友好，轻量 |
| **路由** | Vue Router | 4.x | 标准选择 |
| **HTTP 客户端** | Axios | 1.x | 拦截器机制成熟，便于统一鉴权和错误处理 |
| **后端框架** | Django + DRF | Django 5.x + DRF 3.15 | Admin 开箱即用覆盖后台管理需求，DRF 写 REST API 高效 |
| **后端语言** | Python | 3.12+ | 生态丰富，AI 审核集成方便 |
| **数据库** | PostgreSQL | 16 | 单库事务一致性，jsonb 支持半结构化数据，全文搜索能力强 |
| **缓存/限流** | Redis | 7.x | 缓存热点数据、频率限制、推荐分计数 |
| **对象存储** | 阿里云 OSS / 本地开发用 MinIO | - | 图片存储，CDN 加速 |
| **AI 审核** | Claude API / 通义千问 API | - | 文本 + 图片多模态审核 |
| **任务队列** | Celery + Redis（作为 Broker） | 5.x | 异步 AI 审核、定时自毁、投票截止等后台任务 |
| **部署** | Docker + Docker Compose | - | 一键本地启动，生产可扩展至 K8s |
| **反向代理** | Nginx | - | 静态资源托管、API 转发、HTTPS |

### 3.2 关键设计决策

| 决策点 | 方案 | 说明 |
|--------|------|------|
| 路由模式 | History Mode | Nginx 配置 fallback 到 index.html |
| 列表加载 | 滚动触底加载（Intersection Observer） | 移动端体验优于分页 |
| 深色模式 | CSS Variables + `prefers-color-scheme` | 参考功能设计 13.1，localStorage 持久化 |

| 图片上传 | 前端压缩（compressorjs）→ 直传 OSS | 减轻服务端负担 |
| 移动端适配 | Vant 4 + 媒体查询 | 移动优先，桌面端双列 |

### 5.2 关键设计说明

| 设计点 | 说明 |
|--------|------|
| **身份隔离** | `Post`/`Comment` 通过 `identity_id` 关联 `AnonymousIdentity`，API 层只返回匿名信息，永远不暴露 `user_id` |
| **通用点赞** | `Like` 表使用 `target_type`（post/comment）+ `target_id` 实现多态关联，一张表搞定帖子和评论点赞 |
| **通用举报** | `Report` 同理，`target_type` 区分帖子/评论/私信 |
| **评论嵌套** | `Comment.parent_id` 自引用实现回复层级，API 返回时组装为树形结构 |
| **内容状态** | Post/Comment 的 `status` 字段：`normal` / `ai_suspect` / `pending` / `rejected`，贯穿三级审核 |
| **联合唯一约束** | `Like(user_id, target_type, target_id)` 唯一、`Favorite(user_id, post_id)` 唯一，数据库层面防重复 |
| **软删除** | 帖子删除采用 `is_deleted` 标记，保留数据供管理员追溯 |

## 七、Redis 缓存策略

| 缓存键 | 用途 | 过期时间 | 更新策略 |
|--------|------|---------|---------|
| `post:hot:list` | 热门帖子 ID 列表 | 5 分钟 | 定时重算 |
| `post:{id}:counts` | 单帖点赞/评论/收藏计数 | 10 分钟 | 写时更新 + 异步回写 DB |
| `user:{id}:unread` | 未读消息计数 | 无过期 | 新通知时 INCR，标已读时重算 |
| `user:{id}:tag_pref` | 用户标签偏好分 | 1 小时 | 交互行为时更新 |
| `rate:post:{user_id}` | 发帖频率限制 | 1 分钟 | 发帖时 INCR |
| `rate:post:daily:{user_id}` | 每日发帖计数 | 当天结束 | 发帖时 INCR |
| `rate:search:{ip}` | 搜索频率限制 | 1 分钟 | 搜索时 INCR |
| `jwt:blacklist:{jti}` | JWT 黑名单（登出） | 与 Token 同过期 | 登出时 SET |

## 八、三级审核实现方案

```
                用户提交内容
                     │
                     ▼
            ┌─────────────────┐
            │  L1: DFA 敏感词  │  ◄── 同步，< 5ms
            │  (Python 实现)   │
            └────────┬────────┘
                     │
            ┌────────┴────────┐
            │                 │
       命中硬拦截          通过/软标记
       → 400 返回          → 内容入库（status=normal）
       提示修改                │
                              ▼
                    ┌──────────────────┐
                    │  Celery 异步任务   │
                    │  L2: AI 审核      │  ◄── 异步，1-3s
                    └────────┬─────────┘
                             │
                    ┌────────┴────────┐
                    │                 │
                high 风险         safe/low
                → status=pending   → 保持 normal
                  隐藏内容            │
                    │            medium
                    │            → status=ai_suspect
                    │              正常展示，等人工复查
                    ▼
            ┌──────────────────┐
            │  L3: Django Admin │  ◄── 人工审核队列
            │  审核操作面板      │
            └──────────────────┘
```

**DFA 敏感词实现：** 在 `moderation/services.py` 中用 Trie 树构建 DFA 自动机，应用启动时从数据库加载词库到内存，管理后台修改词库时通过 Django Signal 触发重建。

**AI 审核实现：** Celery Task 调用 Claude/通义千问 API，Prompt 要求返回结构化 JSON `{ risk_level, reason, category }`。图片审核使用多模态能力。超时或异常时 fallback 到人工队列。

## 九、推荐系统实现方案

```
用户行为（点赞/评论/收藏/浏览）
         │
         ▼
┌─────────────────────┐
│  实时更新标签偏好分   │  ◄── Django Signal / View 层触发
│  Redis HINCRBY       │
│  异步回写 DB          │
└─────────┬───────────┘
          │
          ▼  请求推荐列表时
┌─────────────────────┐
│  计算推荐分           │
│                     │
│  推荐分 =            │
│    标签匹配度 × 0.4  │  ◄── 用户标签偏好权重（归一化 0-1）
│  + 热度分    × 0.3  │  ◄── (赞×2+评论×3+收藏×1) 归一化
│  + 时间衰减  × 0.3  │  ◄── 1/(1+小时数/24)
│                     │
│  混入 20-30% 随机内容 │  ◄── 防信息茧房
└─────────┬───────────┘
          │
          ▼
    返回排序后的帖子列表
```

**冷启动处理：**
- 新用户：无偏好数据时退化为热门排序
- 新帖子：发布 2 小时内，时间衰减分额外 +0.2 曝光加成
